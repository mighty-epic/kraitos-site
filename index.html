Why Kraitos Exists

LLMs can reason about work. Running that work on a live machine is a systems problem.

Kraitos focuses on the layer between the model and the computer: what the system can observe, which tools it can call, how it recovers when a path fails, and what evidence it leaves behind for the human owner.

Operator Loop

Observe, act, verify.

01

Observe

Browser snapshots, screenshots, OCR, logs, and memory give the loop current state before it acts.

02

Act

Kraitos can use file tools, shell commands, browser refs, a Chrome bridge, desktop input, and jobs.

03

Verify

Each step feeds evidence back into the loop so it can continue, recover, or stop with a readable result.

Built For Self-Hosting

Automation without surrendering the runtime.

Kraitos is designed for people who want inspectable automation across local, remote, and VPS machines.

  • Plain Markdown memory
  • Explicit authorized user controls
  • Inspectable logs and screenshots
  • Local, remote, and VPS deployment paths
  • Telegram and desktop control surfaces

Browser Automation

Less guesswork, more state.

Kraitos uses structured browser snapshots with stable references instead of relying only on raw coordinates. When standard automation is brittle, a Chrome extension bridge can connect to a real session. When the DOM is not enough, OCR and desktop input remain available.

FAQ

The short version.

Is Kraitos an autonomous employee?

No. It is a self-hosted operator layer for controlled execution, observable state, and user-owned infrastructure.

Why not just use Selenium?

Some tasks work with Selenium. Others need a real logged-in browser session, page refs, or desktop-level fallback.

Where is memory stored?

The project uses human-readable memory files such as Markdown logs rather than hiding all state in an opaque service.

What should I try first?

Clone the repo, configure a model provider, run a control surface, and test a small browser task where logs are visible.
